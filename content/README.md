# コンテンツMarkdown執筆ガイド

このディレクトリには、スライドに変換したい調査・執筆内容のMarkdownファイルを配置します。

---

## 📋 概要

コンテンツMarkdown（以下、コンテンツMD）は、調査・研究・執筆した内容をまとめたMarkdownファイルです。AIがこのファイルを読み込んで、Slidev形式のスライドに自動変換します。

**目的**:
- 調査・執筆とスライド作成を分離
- 長文のコンテンツを構造化してスライドに変換
- 一貫性のあるスライド生成

---

## 🎯 スライド変換しやすいMarkdown構造

### 基本ルール

1. **見出しレベル**: スライドの構造を決定
2. **段落**: 1段落 = 1つの主張/概念
3. **リスト**: 箇条書き（最大7項目）
4. **コードブロック**: 言語指定必須
5. **画像**: 説明的なalt属性

---

## 📝 見出しレベルとスライドの対応

### H1 (`#`) - セクション区切り

```md
# セクションタイトル
```

**変換後**:
- セクション区切りスライド
- `layout: center` + `slide-gradient-bg`
- 大きな章の開始に使用

**使用シーン**:
- 新しいトピックの開始
- プレゼンテーションのパート分け
- 休憩の区切り

---

### H2 (`##`) - 新しいスライド

```md
## スライドのタイトル

本文内容...
```

**変換後**:
- 新しいスライド（1ページ）
- H2がスライドのタイトル
- 本文が内容

**使用シーン**:
- 各トピックの詳細説明
- 通常のコンテンツスライド

---

### H3 (`###`) - サブセクション

```md
## スライドタイトル

### サブトピック

内容...
```

**変換後**:
- 同じスライド内のサブセクション
- サブトピックとして表示

**使用シーン**:
- 1つのスライド内で複数のポイント
- 階層的な情報

## 🔧 要素別の書き方

### 1. リスト

#### Good（推奨）

```md
## 主な特徴

- シンプルで直感的
- 高速なビルド
- 拡張性が高い
- デザインカスタマイズ可能
```

**変換後**:
- `EmojiText` で装飾される可能性あり
- 最大7項目まで（7±2の法則）

#### Bad（非推奨）

```md
## 主な特徴

- 項目1項目1項目1項目1項目1項目1項目1項目1（長すぎる）
- 項目2
- 項目3
...
- 項目10（多すぎる）
```

---

### 2. コードブロック

#### Good（推奨）

````md
## TypeScript の型定義

TypeScriptでは型を定義できます：

```ts
interface User {
  name: string
  age: number
}
```

この型定義により、安全なコードを書けます。
````

**変換後**:
- 2カラムレイアウト
  - left: 説明文
  - right: コードブロック

#### Bad（非推奨）

````md
```ts
// コードだけでコンテキストなし
const x = 123
```
````

**ルール**:
- コードブロックには必ず言語指定（`ts`, `js`, `py`, など）
- コードの前後に説明を追加
- コードが長い場合は複数スライドに分割を検討

---

### 3. 画像

#### Good（推奨）

```md
## システム構成図

システムの全体像は以下の通りです：

![システム構成図](./images/architecture.png)

この構成により、スケーラブルなシステムを実現しています。
```

**変換後**:
- `CenteredImage` コンポーネント
- キャプション付き

#### Bad（非推奨）

```md
![](./image.png)
```

**ルール**:
- 説明的なalt属性を必ず付ける
- 画像の前後に説明文を追加
- 相対パスまたはURL

---

### 4. 引用

```md
> 重要な引用やキーメッセージ
```

**変換後**:
- blockquote として強調表示

---

### 5. 表

#### Good（シンプルな表）

```md
| 項目 | 説明 |
|------|------|
| A | 説明A |
| B | 説明B |
| C | 説明C |
```

**変換後**:
- そのまま表示

#### Bad（複雑な表）

```md
| A | B | C | D | E | F | G |
|---|---|---|---|---|---|---|
| ... 多くのカラム ... |
```

**ルール**:
- 列数は最大5列まで
- 行数は最大7行まで
- 複雑な表は複数スライドに分割

---

## 📐 スライド分割のルール

### 1スライド = 1つの主張/概念

**Good（1つの主張）**:

```md
## React Hooks のメリット

- 状態管理がシンプル
- ロジックの再利用が容易
- クラスコンポーネント不要
```

**Bad（複数の主張を詰め込む）**:

```md
## React HooksとReduxとContext APIとRecoilと...

（10個以上の概念を1スライドに詰め込む）
```

---

### 7±2の法則

1スライドの情報量:
- リスト項目: 最大7項目
- 段落: 2-3段落まで
- コード行数: 20行以内

**長いコンテンツは自動分割**:

```md
## 長いリスト

- 項目1
- 項目2
- 項目3
- 項目4
- 項目5
- 項目6
- 項目7
- 項目8  ← ここから次のスライドに分割
- 項目9
- 項目10
```

## 🎨 表紙情報の指定

### Frontmatter（推奨）

```md
---
title: プレゼンテーションタイトル
subtitle: サブタイトルや簡単な説明
event: イベント名
author: あなたの名前
social:
  github: username
  twitter: username
---

# 本文開始
```

**または、最初のH1**:

```md
# プレゼンテーションタイトル

サブタイトルや簡単な説明

著者: あなたの名前
```

## 💡 Tips

### Tip 1: セクションを意識

H1で大きな区切りを作ると、聴衆が全体像を把握しやすくなります。

```md
# 第1部: 背景
## トピック1
## トピック2

# 第2部: 提案
## トピック3
## トピック4
```

### Tip 2: ストーリーを作る

スライドは物語として構成すると効果的：

1. 導入（背景・問題提起）
2. 展開（解決策・提案）
3. 結論（結果・まとめ）

### Tip 3: ビジュアルを活用

- 図や画像を積極的に使用
- コードは視覚的に分かりやすく
- 表は見やすいサイズに

### Tip 4: シンプルに保つ

- 1スライド = 1メッセージ
- 詰め込みすぎない
- 余白を活用
